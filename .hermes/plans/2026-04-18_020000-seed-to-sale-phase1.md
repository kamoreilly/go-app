# Implementation Plan: Seed-to-Sale Platform — Phase 1

## Goal

Build a regulatory-credible seed-to-sale cannabis cultivation management platform (Phase 1 only). This is a full-stack Go + React application with PostgreSQL, implementing facility scoping, cultivation tracking, plant lineage, and harvest workflows.

**Tech Stack:**
- Backend: Go 1.26, **Fiber v3**, **GORM** (PostgreSQL), JWT auth
- Frontend: React 19, **TanStack Start**, Tailwind CSS 4, shadcn/ui
- Database: PostgreSQL 17
- Ports: Backend `:3000`, Frontend `:3001`

## Phase 1 Database Schema (Migration 000001)

### Core Entities

1. **facilities** — `id (uuid pk)`, `name`, `address`, `timezone`, `canopy_sqft`, `plant_limit`, `created_at`, `updated_at`, `version (optimistic lock)`
2. **licenses** — `id (uuid pk)`, `facility_id (fk)`, `license_number`, `license_type`, `jurisdiction`, `issued_at`, `expires_at`, `status`, `metadata (jsonb)`
3. **zones** — `id (uuid pk)`, `facility_id (fk)`, `name`, `zone_type (enum: veg, flower, propagation, drying, quarantine)`, `canopy_sqft`, `created_at`
4. **strains** — `id (uuid pk)`, `facility_id (fk)`, `name`, `genetics`, `thc_percent`, `cbd_percent`, `flowering_time_days`, `created_at`
5. **mother_plants** — `id (uuid pk)`, `facility_id (fk)`, `strain_id (fk)`, `planted_at`, `status (enum: active, retired, destroyed)`, `notes`, `created_at`
6. **plants** — `id (uuid pk)`, `facility_id (fk)`, `strain_id (fk)`, `zone_id (fk)`, `mother_id (fk nullable)`, `planted_at`, `stage (enum: propagation, vegetative, flowering, harvesting, dried, packaged)`, `state (enum: active, harvested, destroyed)`, `weight_grams`, `batch_id (fk nullable)`, `created_at`, `updated_at`
7. **harvest_batches** — `id (uuid pk)`, `facility_id (fk)`, `strain_id (fk)`, `zone_id (fk)`, `harvested_at`, `total_weight_grams`, `wet_weight`, `dry_weight`, `waste_weight`, `moisture_loss`, `status (enum: wet, curing, cured, released, quarantined)`, `lab_test_id (fk nullable)`, `notes`, `created_at`, `updated_at`
8. **batch_plants** — `id (uuid pk)`, `batch_id (fk)`, `plant_id (fk)`, `plant_weight_grams`
9. **harvest_mass_balance** — `id (uuid pk)`, `batch_id (fk)`, `input_weight`, `output_weight`, `waste_weight`, `moisture_loss`, `adjustment_weight`, `adjustment_reason`, `created_at`

### State Machines

- Plant.stage: `propagation → vegetative → flowering → harvesting → dried → packaged`
- Plant.state: `active | harvested | destroyed`
- HarvestBatch.status: `wet → curing → cured → released | quarantined`

### Immutable vs Soft-Deletable

- Immutable: `harvest_batches`, `harvest_mass_balance`, `batch_plants` (once created, never modified — only adjustments added)
- Soft-deletable: `facilities`, `zones`, `strains`, `plants`, `mother_plants`
- Licenses: never deleted, only `status` changed to `expired` or `revoked`

### Timestamps

- All stored as `TIMESTAMPTZ` (UTC) in PostgreSQL
- Application layer converts to facility timezone for display

## Phase 1 API Endpoints

All endpoints prefixed with `/v1`. Facility-scoped resources require `X-Facility-ID` header.

### Auth
- `POST /v1/auth/register` — Register user
- `POST /v1/auth/login` — Login, returns JWT

### Facilities
- `GET /v1/facilities` — List user's facilities
- `POST /v1/facilities` — Create facility
- `GET /v1/facilities/:id` — Get facility
- `PUT /v1/facilities/:id` — Update facility (with version for optimistic locking)
- `DELETE /v1/facilities/:id` — Soft delete

### Licenses
- `GET /v1/facilities/:id/licenses` — List licenses
- `POST /v1/facilities/:id/licenses` — Add license
- `PUT /v1/licenses/:id` — Update license
- `GET /v1/licenses/expiring` — List expiring licenses (within 30 days)

### Zones
- `GET /v1/facilities/:id/zones` — List zones
- `POST /v1/facilities/:id/zones` — Create zone
- `PUT /v1/zones/:id` — Update zone
- `DELETE /v1/zones/:id` — Soft delete

### Strains
- `GET /v1/facilities/:id/strains` — List strains
- `POST /v1/facilities/:id/strains` — Create strain
- `PUT /v1/strains/:id` — Update strain
- `DELETE /v1/strains/:id` — Soft delete

### Mother Plants
- `GET /v1/facilities/:id/mothers` — List mothers
- `POST /v1/facilities/:id/mothers` — Create mother plant
- `GET /v1/mothers/:id` — Get mother plant
- `PUT /v1/mothers/:id` — Update mother plant
- `POST /v1/mothers/:id/clone` — Take clone from mother

### Plants
- `GET /v1/facilities/:id/plants` — List plants (filterable by stage, zone, strain)
- `POST /v1/facilities/:id/plants` — Create plant
- `GET /v1/plants/:id` — Get plant with lineage
- `PUT /v1/plants/:id` — Update plant (stage transition, zone move)
- `POST /v1/plants/:id/transition` — State machine transition
- `POST /v1/plants/:id/destroy` — Destroy plant
- `POST /v1/plants/bulk-move` — Move multiple plants between zones

### Harvest Batches
- `GET /v1/facilities/:id/batches` — List batches
- `POST /v1/facilities/:id/batches` — Create batch from plants
- `GET /v1/batches/:id` — Get batch with mass balance
- `PUT /v1/batches/:id/mass-balance` — Record mass balance (immutable once finalized)
- `POST /v1/batches/:id/transition` — Batch status transition
- `GET /v1/batches/:id/lineage` — Forward/backward traceability

## Risks and Open Questions

1. **Migration strategy**: The existing SQLite `test.db` and simple `internal/` structure will be replaced entirely. Decide whether to preserve backward compat or just start fresh. → **Start fresh**: Phase 1 rewrites the backend architecture from scratch.
2. **GORM vs raw SQL**: GORM v2 with PostgreSQL is standard. Use `*gorm.DB` injection via struct fields, not global singletons.
3. **Frontend routing**: TanStack Start uses file-based routing with auto-generated `routeTree.gen.ts`. All route files must follow the naming convention.
4. **Optimistic locking**: `version` column + `WHERE version = ?` on updates. Middleware or service layer handles it.
5. **State machines**: Enforced in service layer, not DB. Service validates transitions before executing.
6. **Timezone handling**: Store UTC, render in facility timezone. Use `facility.Timezone` field from DB.
